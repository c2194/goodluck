# minpng.py 功能流程分析

## 1. 文件定位

minpng.py 是 up.php 上传流程中的图片后处理脚本，负责将前端生成的三色墨水屏 PNG 转换为最小体积的 3 色调色板 PNG，供 ESP32 等嵌入式设备下载使用。

### 1.1 在上传流程中的位置

```
前端上传 PNG → up.php 保存原图 → minpng.py 原地压缩覆盖 → 设备拉取最终文件
```

up.php 中的调用方式：

```php
$cmdImg = escapeshellcmd('python3') . ' ' . escapeshellarg($minpngScript)
        . ' ' . escapeshellarg($imagePath) . ' ' . escapeshellarg($imagePath);
```

注意输入和输出路径相同，即原地覆盖。

## 2. 命令行接口

```
python minpng.py <input> <output>
```

| 参数 | 说明 |
|------|------|
| `input` | 输入图片路径（任意 Pillow 支持的格式） |
| `output` | 输出 PNG 路径（可与 input 相同） |

脚本会自动创建输出路径的父目录（`mkdir(parents=True, exist_ok=True)`）。

## 3. 核心算法

### 3.1 目标调色板

脚本定义了固定的 3 色调色板：

| 索引 | 颜色 | RGB 值 |
|------|------|--------|
| 0 | 白色 | (255, 255, 255) |
| 1 | 黑色 | (0, 0, 0) |
| 2 | 红色 | (255, 0, 0) |

这 3 种颜色对应三色墨水屏的物理显示能力。

### 3.2 最近邻量化

对每个像素，计算它与 3 种目标颜色的欧氏距离平方，选择距离最小的颜色索引：

```
distance² = (R₁-R₂)² + (G₁-G₂)² + (B₁-B₂)²
```

使用平方距离（不开根号）是因为只需要比较大小关系，省去 sqrt 运算。

### 3.3 调色板 PNG 输出

不使用 RGB 模式，而是创建 P 模式（paletted）图片：

1. 创建空的 P 模式图像。
2. 设置 3 色调色板（补零到 256 色共 768 字节）。
3. 每个像素存储的是 0/1/2 索引值而非完整 RGB。
4. 保存时启用 `optimize=True`，让 Pillow 选择最优的 PNG 压缩参数。

## 4. 完整处理流程

```
输入图片
  │
  ▼
① Image.open() 打开图片
  │
  ▼
② .convert("RGB") 统一转为 RGB 模式
  （处理 RGBA、L、P 等其他模式的输入）
  │
  ▼
③ 创建同尺寸 P 模式空图
  │
  ▼
④ 设置 3 色调色板 + 补零到 256 色
  │
  ▼
⑤ 逐像素量化
  ├─ 读取原图 (r, g, b)
  ├─ 计算与白/黑/红的距离²
  └─ 写入最近颜色的索引 (0/1/2)
  │
  ▼
⑥ 保存为 PNG（optimize=True）
  │
  ▼
⑦ 输出压缩统计（输入大小、输出大小、节省比例）
```

## 5. 函数说明

### 5.1 color_distance_sq(c1, c2)

计算两个 RGB 颜色之间的欧氏距离平方。返回 int 类型，避免浮点运算。

### 5.2 nearest_palette_index(r, g, b)

对给定像素遍历 3 种目标颜色，返回距离最小的颜色索引（0=白、1=黑、2=红）。

### 5.3 minimize_tricolor_png(input_path, output_path)

核心处理函数：

1. 打开输入图片并转为 RGB。
2. 创建 P 模式输出图。
3. 设置调色板。
4. 逐像素量化。
5. 保存为优化的 PNG。

### 5.4 main()

入口函数：

1. 解析命令行参数。
2. 解析为绝对路径。
3. 确保输出目录存在。
4. 调用 minimize_tricolor_png。
5. 打印压缩前后的文件大小和节省比例。
6. 返回 0 表示成功。

## 6. 压缩效果分析

### 6.1 为什么能大幅缩小

前端 canvas.toBlob('image/png') 生成的是 RGBA 32 位 PNG，每个像素 4 字节。而转为 P 模式后：

- 每个像素只占调色板索引的位数。3 色只需要 2 bit，但 PNG P 模式最小是 1 byte/pixel 索引。
- 调色板只有 3 色，PNG 滤波器和 DEFLATE 压缩对低色彩图片效率极高。
- 去掉了 Alpha 通道。

典型场景下，一张 270×400 的三色图：
- 前端原始 PNG：约 50-150 KB
- 压缩后：约 5-20 KB

### 6.2 不做抖动

脚本使用硬量化（nearest neighbor），不做 Floyd-Steinberg 抖动。这是因为输入图片本身已经在前端经过了三色抖动处理（tricolor.js），像素值应当已经非常接近三种目标色。这一步只是做格式转换和"清理"，不需要再做视觉优化。

## 8. 注意事项

### 8.1 逐像素循环的性能

Python 逐像素 for 循环处理 270×400 = 108,000 像素，在服务端速度通常在几十毫秒到几百毫秒之间。对于当前业务场景（单次上传、低并发）完全可接受。如果未来需要处理更大图片或更高并发，可以用 NumPy 向量化重写量化逻辑。

### 8.2 输入图片不限格式

虽然 up.php 传入的是 PNG，但 `Image.open()` + `.convert("RGB")` 可以处理 Pillow 支持的任意图片格式（JPEG、BMP、WebP 等）。

### 8.3 原地覆盖安全性

up.php 调用时输入输出路径相同。Pillow 的 `Image.save()` 会先写入再关闭，而 `Image.open()` 使用 with 语句确保读取完成。由于量化过程在内存中进行（创建了新的 out 图像对象），所以原地覆盖是安全的。

### 8.4 import math 未使用

文件顶部 `import math` 但代码中未使用（距离计算直接用了整数运算），属于残留导入。
