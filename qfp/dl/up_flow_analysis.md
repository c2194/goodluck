# up.php 功能流程分析

## 1. 文件定位

up.php 是祝福牌系统的服务端上传接口，负责接收前端提交的图片和可选音频，保存到对应设备目录，并调用后处理脚本完成压缩与格式转换。

## 2. 请求格式

- 方法：POST
- Content-Type：multipart/form-data
- 查询参数：直接拼在 URL 上，格式为 `?YYYYMMMMMMMMM KKKKKKKK`（21 位字符串）

### 2.1 查询参数结构

正则：`/^(\d{4})([A-Za-z0-9]{9})([A-Za-z0-9]{8})$/`

| 分组 | 长度 | 含义 | 示例 |
|------|------|------|------|
| `$matches[1]` — monthYear | 4 位数字 | 年月标识，对应存储目录名 | `0226`、`0326` |
| `$matches[2]` — mac62 | 9 位字母数字 | 设备标识（Base62 编码的 MAC 地址），对应子目录名 | `ae9TUVYwK` |
| `$matches[3]` — key | 8 位字母数字 | 祝福位标识，对应 config.json 中的键 | `OytfTjKN` |

### 2.2 表单字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `image` | file (PNG) | 是 | 三色墨水屏图片 |
| `audio` | file (WAV/WebM) | 否 | 录制的祝福语音频 |

## 3. 目录与文件结构

```
dl/
├── up.php                    ← 本文件
├── minpng.py                 ← 图片压缩脚本
├── enc_au_pic.py             ← 音频转换脚本
├── 0226/                     ← monthYear 目录
│   ├── ae9TUVYwK/            ← 设备 mac62 目录
│   │   ├── config.json       ← 设备配置（含所有 key 的 state）
│   │   ├── OytfTjKN5.png     ← key + state 版本号的图片
│   │   ├── OytfTjKN5.mp3     ← 对应音频（转换后）
│   │   └── ...
│   └── ...
└── 0326/
    └── ...
```

### 3.1 config.json 结构

```json
{
    "OytfTjKN": {
        "pw": "",
        "state": "5"
    },
    "iIjPGidc": {
        "pw": "",
        "state": "12"
    }
}
```

- 每个 key 代表设备上的一个祝福位。
- `state` 是字符串类型的整数，每次上传自增 1。
- `pw` 字段当前为空，预留用途（可能是密码保护）。

## 4. 完整处理流程

```
请求进入
  │
  ▼
① 解析查询参数
  ├─ 空参数 → 400「缺少参数」
  └─ 正则不匹配 → 400「参数格式不正确」
  │
  ▼
② 定位设备目录
  ├─ 拼接路径：__DIR__ / monthYear / mac62
  ├─ 目录不存在 → 404「目标目录或配置不存在」
  └─ config.json 不存在 → 404
  │
  ▼
③ 读取并校验 config.json
  ├─ JSON 解码失败 → 500「配置文件损坏」
  └─ key 不在配置中 → 403「无效的 key」
  │
  ▼
④ 计算新版本号
  └─ newState = 当前 state + 1
  │
  ▼
⑤ 保存图片
  ├─ 检查 $_FILES['image'] 是否上传成功
  │   └─ 失败 → 400「图片上传失败」
  ├─ 文件名：{key}{newState}.png
  └─ move_uploaded_file 失败 → 500「保存图片失败」
  │
  ▼
⑥ 压缩图片（minpng.py）
  ├─ 调用：python3 minpng.py <图片路径> <图片路径>（原地覆盖）
  ├─ 成功 → 继续
  └─ 失败 → 记录日志，不中断流程
  │
  ▼
⑦ 保存音频（可选）
  ├─ 如果 $_FILES['audio'] 存在且上传成功
  │   ├─ 文件名：{key}{newState}.wav
  │   └─ move_uploaded_file 失败 → 500「保存音频失败」
  └─ 未上传 → 跳过
  │
  ▼
⑧ 转换音频（enc_au_pic.py，可选）
  ├─ 调用：python3 enc_au_pic.py <wav路径>
  ├─ 脚本内部用 ffmpeg 转换为 MP3（16kbps 单声道 16kHz）
  ├─ 转换成功后删除原始 WAV，保留 .mp3
  ├─ 成功 → 继续
  └─ 失败 → 记录日志，不中断流程
  │
  ▼
⑨ 更新 config.json
  └─ 将 key 的 state 写为 newState
  │
  ▼
⑩ 返回结果
  └─ 200「上传成功」+ 图片状态 + 音频状态 + 客户端日志脚本
```

## 5. 辅助函数

### 5.1 add_client_log(message)

将日志消息追加到全局数组 `$clientLogs`。这些日志最终会被嵌入到响应的 `<script>` 标签中，让前端浏览器控制台也能看到服务端处理结果。

### 5.2 respond(message, statusCode)

统一响应函数：

1. 设置 HTTP 状态码。
2. 输出 HTML 转义后的消息文本。
3. 如果有客户端日志，追加一段 `<script>` 输出 `console.log()`。
4. 立即 `exit` 终止脚本。

## 6. 外部依赖

### 6.1 minpng.py — 三色 PNG 压缩

- 功能：把任意 PNG 量化为 3 色调色板图（白 / 黑 / 红），生成最小体积的 paletted PNG。
- 调用方式：`python3 minpng.py <输入路径> <输出路径>`（支持原地覆盖）。
- 依赖：Pillow。
- 原理：逐像素计算与三个目标颜色的欧氏距离，选最近色，写入 P 模式图片。

### 6.2 enc_au_pic.py — 音频转 MP3

- 功能：将 WAV（或浏览器录制的 WebM）转换为低码率 MP3。
- 调用方式：`python3 enc_au_pic.py <wav路径>`。
- 依赖：系统安装的 ffmpeg。
- 参数：单声道、16kbps、16kHz 采样率、LAME 编码。
- 副作用：转换成功后删除原始 WAV 文件。

## 7. 错误处理策略

接口的错误处理分两个级别：

### 7.1 致命错误（中断请求）

| 条件 | HTTP 状态码 | 消息 |
|------|-------------|------|
| 缺少查询参数 | 400 | 缺少参数 |
| 参数格式不匹配 | 400 | 参数格式不正确 |
| 设备目录不存在 | 404 | 目标目录或配置不存在 |
| config.json 解码失败 | 500 | 配置文件损坏 |
| key 不在配置中 | 403 | 无效的 key |
| 图片未上传或上传错误 | 400 | 图片上传失败 |
| 图片保存失败 | 500 | 保存图片失败 |
| 音频保存失败 | 500 | 保存音频失败 |

### 7.2 非致命错误（记录但继续）

- 图片压缩失败：记录 error_log 和客户端日志，继续执行。
- 音频转换失败：记录 error_log 和客户端日志，继续执行。

这种设计确保图片至少能保存原始版本，后处理失败不会导致整个上传流程回滚。

## 8. 版本控制机制

每次上传不会覆盖旧文件，而是通过 state 自增产生新文件名：

- 第 1 次上传：`OytfTjKN1.png`
- 第 2 次上传：`OytfTjKN2.png`
- 第 5 次上传：`OytfTjKN5.png`

旧版本文件仍然保留在目录中，不会被删除。这意味着：

- 可以回溯历史版本。
- 磁盘空间会持续增长，没有自动清理机制。

## 9. 注意事项

### 9.1 config.json 并发写入风险

多个请求同时写入同一个 config.json 时，`file_get_contents` → 修改 → `file_put_contents` 不是原子操作，可能导致数据丢失。当前场景下并发概率低（单设备单用户），但如果流量增大需要加文件锁（`flock`）。

### 9.2 文件名中的 state 无补零

state 直接拼接为字符串（如 `1`、`12`），不做补零。这导致文件排序在个位数和多位数之间不连续（`1.png` 排在 `12.png` 前面而不是 `2.png` 前面）。

### 9.3 音频 MIME 类型与扩展名

前端 MediaRecorder 在不同浏览器上产出的格式可能是 audio/webm 或 audio/ogg，但文件固定保存为 .wav。enc_au_pic.py 通过 ffmpeg 处理，ffmpeg 能自动检测输入格式，所以实际上能兼容多种输入。但严格来说文件扩展名与实际内容不一致。

### 9.4 图片压缩是原地覆盖

minpng.py 的输入和输出路径相同，意味着原始上传的 PNG 会被压缩后的 3 色调色板 PNG 替换，无法保留原始未压缩版本。
